# CLAUDE.md — FietsNav project context

Personal, Google-Maps-style **bike navigation** web app for the Netherlands. It routes along
**real cycle paths, ferries and the knooppunten node network** — not car roads. No build step,
no backend, no framework: just static files. Read `HANDOVER.md` for the full picture (running,
phone simulators, going native). This file is the quick always-on context.

## Stack
- Vanilla **HTML + CSS + JS (ES2020)**, single-page. No bundler, no transpile.
- **Leaflet 1.9.4** (CDN) for the map.
- All logic in `app.js`; all markup + CSS in `index.html`. Keep it that way unless there's a strong reason.

## External services (all free, public, called from the browser)
- **Routing** — BRouter `https://brouter.de/brouter` (GeoJSON). Profiles used: `trekking`
  (Recommended), `fastbike` (Fast), `safety` (Quiet), `shortest`. CORS: `*`.
- **Search** — Photon `https://photon.komoot.io` (type-ahead, NL-biased). CORS enabled.
- **POIs** — Overpass `https://overpass-api.de/api/interpreter` (knooppunten via `rcn_ref`,
  bike parking, ferries, stations). Only queried at zoom ≥ 12.
- **Tiles** — CyclOSM (default), CARTO light, OSM standard; waymarkedtrails cycling overlay.

## File map
- `index.html` — layout, CSS (design tokens in `:root`, dark overrides in `[data-theme="dark"]`),
  DOM the JS hooks into. Structure: `#map`, `#dock`(→`#search` + `#sheet`), `#fabs`, `#pop`,
  `#nav`/`#navbar`, `#drawer`, `#toast`.
- `app.js` — everything: map, geocoding+autocomplete, routing, turn computation, elevation/surface
  profile, navigation, POIs, saved places + recents, theme, bottom sheet, UI wiring. Commented sections.
- `sw.js` / `manifest.webmanifest` / `icon-*.png` — PWA. SW = stale-while-revalidate shell + capped tile cache.
- `capacitor.config.json` / `NATIVE.md` — native build (Capacitor) config + instructions; `www/` is generated by `native:prep`.
- `test/smoke.test.js` — pure-logic unit tests (no browser/network needed). Currently 27 checks.
- `package.json` — `dev`, `test`, `check`, `native:prep`, `cap:*` scripts.

## Layout (map-first, like Google/Apple Maps)
- **Mobile (≤760px):** frosted floating `#search` bar on top; `#results` lives in a draggable
  `#sheet` (peek ↔ full, dragged via `#grab`). Sheet is hidden when there's no route (map-first).
- **Desktop (≥760px):** `#dock` becomes a left rail — `#search` (pinned height) above `#sheet`
  (`flex:1`, scrolls). Don't let either collapse the other (search must be `flex:0 0 auto`).
- Theme: `#themeBtn` toggles; drawer has Auto/Light/Dark. Dark also pairs the map (`light`↔`dark` base).
  Theme is resolved before paint by an inline script in `<head>` (no flash).

## Commands
- `npm run dev` — serve at http://localhost:8000 (localhost = secure context, so GPS works).
- `npm test` — run the smoke tests (must stay green).
- Going native: see `HANDOVER.md` → "Make it an actual phone app".

## Key design decisions (don't regress these)
- **Turn-by-turn directions are computed from the route geometry** (RDP corner detection +
  bearing deltas in `buildTurns`/`classifyTurn`), NOT from BRouter's internal voicehint codes —
  this is deliberate for robustness. Street names are a *best-effort* enrichment parsed from
  BRouter `messages` (`namesFromMessages`); never let missing names break anything.
- **The distance comparison baseline is the *shortest bike route*** (BRouter `shortest`
  profile via `shortestBikeRoute`), i.e. the more direct Google-Maps-style cycling route — NOT a
  car route. The card shows how much extra distance the scenic/quiet route adds. Its time is
  estimated at the chosen route's cycling speed — BRouter's `shortest` profile reports an
  unrealistically slow (walking-like) time, so we never surface that raw value.
- **Defensive parsing everywhere**: route stats, names, POIs, and geocoding all fall back
  gracefully if a field is missing or an API hiccups.
- Voice via the Web Speech API (`speak`); off-route reroute in `reroute`/`snapToRoute`.
- **Elevation profile** (`profileFromMessages`): BRouter's `Elevation` message column is the
  **absolute height in metres** (often single-digit / negative in NL polders), NOT a delta.
  Reject SRTM void sentinels (e.g. `-2048`) via a `>-430 && <5000` clamp, light-smooth the line,
  and enforce a min vertical span so flat routes look flat. The climb figure shown is BRouter's
  smoothed `filtered ascend` (reliable) — descent is NOT shown (no trustworthy source); the card
  shows high/low points instead.
- **Tested-function contract**: `test/smoke.test.js` runs `app.js` in a Proxy sandbox (no real
  DOM/Leaflet). Any new *module-level* code must be Proxy-safe — only touch sandbox globals
  (`L, document, window, navigator, location, localStorage, fetch, innerWidth/Height`) or guard
  others (`AbortController`, `matchMedia`, `URLSearchParams`, `requestAnimationFrame`) with
  `typeof`/`window.`/try-catch. Don't rely on `requestAnimationFrame` for layout — it doesn't fire
  in headless tabs (measure synchronously after setting `innerHTML`).

## Gotchas
- **GPS / live nav need a secure context** — works on `https://` or `localhost`, not bare `file://`.
- Don't use `localStorage` for anything that must survive reinstalls — it's per-browser only
  (saved Home/Work/routes live there by design).
- New BRouter profile? Add a `.chip` in `index.html` and it flows through; unknown profiles
  fall back to `trekking` with a toast.
- Overpass is rate-limited — keep the zoom-12 guard and the debounce on `moveend`.

## UI rules — "always make the best UI"
- Mobile-first and responsive. Breakpoint at **760px**: phone = top search bar + bottom sheet;
  desktop = 380px left rail + full-bleed map. Test both every change.
- Use the existing **design tokens** in `:root` (accent `#0e7c5a`; per-style colours: fast
  `#e8590c`, quiet `#1971c2`, shortest `#7048e8`). Don't introduce one-off colours.
- Touch targets ≥ 44px. Sentence case. One accent, generous whitespace, flat surfaces,
  subtle shadows only where they aid hierarchy.
- After any UI change, **look at it**: device-emulated screenshots at phone + desktop widths
  (see HANDOVER). Don't ship UI you haven't viewed.
- Keep motion subtle and purposeful; never block the map interaction.
